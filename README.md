# ONYX — AI-assisted cash reconciliation (PoC)

A working proof of concept that takes four CSV exports — Trustee account summary, nVision cash balance, nVision transactions, and Axiom open-breaks output — and uses three coordinated agents to parse them, cross-reference the records, classify each break, and draft resolution recommendations.

This is a **single-user PoC**. No auth, no database, no multi-tenancy. Run history is kept in your browser's `localStorage`.

---

## Architecture

```
Browser (index.html + app.js)
  │
  ├─ Login overlay (shown until authenticated)
  │
  ├─ Stage 1 — Orchestration Agent: parse CSVs       [pure JS, in browser]
  ├─ Stage 2 — Data Agent: cross-reference sources   [pure JS, in browser]
  ├─ Stage 3 — Reconciliation Agent: classify breaks ─┐
  ├─ Stage 4 — Orchestration: priority + ranking     │ [pure JS]
  │                                                   │
  ├─ /api/login            ← Password check, returns bearer token
  ├─ /api/classify         ← LLM classification ─────┘
  ├─ /api/draft-resolution ← LLM recommendation text
  └─ /api/draft-email      ← LLM escalation email composition

  All /api/* (except /api/login) require Authorization: Bearer <token>
```

Why this split: only Stages 3 (classification) and the two draft endpoints need an LLM. The deterministic work (parsing, joining, scoring) runs in the browser — it's faster, cheaper, and never burns tokens.

**Secrets never reach the browser.** Both the OpenAI key and the access password live as environment variables on Vercel and are only read inside the `/api/*` functions. The login endpoint compares the submitted password to the env var server-side; the rest of the endpoints check the bearer token on every request.

## Deploy to Vercel — 5-minute path

### 1. Get the code into GitHub
```bash
# from the project root
git init
git add .
git commit -m "Initial ONYX PoC"
git branch -M main
# create a new empty repo at github.com/<you>/onyx, then:
git remote add origin https://github.com/<you>/onyx.git
git push -u origin main
```

### 2. Connect to Vercel
1. Sign in at [vercel.com](https://vercel.com) (GitHub login is easiest).
2. Click **Add New → Project** → import your `onyx` repo.
3. Framework preset: **Other** (Vercel will auto-detect that `/api/*.ts` are serverless functions).
4. Click **Deploy**. First build takes ~1 minute.

### 3. Add environment variables
1. In the Vercel dashboard, open your project → **Settings → Environment Variables**.
2. Add these:

   | Name | Value | Notes |
   |---|---|---|
   | `AZURE_OPENAI_API_KEY` | your Azure OpenAI resource key | Required |
   | `AZURE_OPENAI_ENDPOINT` | e.g. `https://genaitraining-aoai2.openai.azure.com` | Required. Base URL, **without** the trailing `/openai`. |
   | `AZURE_OPENAI_DEPLOYMENT` | e.g. `gpt-4.1` | Defaults to `gpt-4.1` if unset. This is the **deployment name** in your Azure resource, not the OpenAI model family name. |
   | `AZURE_OPENAI_API_VERSION` | e.g. `2025-03-01-preview` | Optional, defaults to that value. |
   | `APP_PASSWORD` | e.g. `ONYX22JUN2026` | Required. The login password. Pick something only you and your reviewers know. |

3. Go to **Deployments** → click the latest → **Redeploy** (so the env vars take effect).

### 4. Cap the spend
Set a budget alert on your Azure OpenAI resource: Azure Portal → your resource → **Cost Management → Budgets**. A $20/month cap with email alerts at 80% is more than enough for daily testing of one user.

That's it. The deployment URL is your live app.

```bash
npm install

# edit .env.local and paste your OPENAI_API_KEY
```

The first time you run `vercel dev` it will link the local folder to a Vercel project — accept the prompts (create a new project if you don't have one yet).

## Using the app

1. Open the deployed URL. You'll see the landing page.
2. Click **Upload Files** (top right, or the hero CTA).
3. Select all four CSVs at once:
   - Trustee cash account summary (`DEALNAME`, `DEALID`, `BEGINNINGBALANCE`, ...)
   - nVision cash balance (`FUNDID`, `CLEARINGACCOUNTID`, `TDCASHBALANCE`, ...)
   - nVision transactions (`CUSIP`, `ASSETID`, `ASSETNAME`, ...)
   - Axiom open-breaks output (`ASOFDATE`, `AMOUNT`, `INPUT`, `DESCRIPTION`, ...)
4. The agents stream their findings in real time. When complete, click **Open queue**.
5. Click any break to open its investigation page. The recommendation text is generated by OpenAI when the page opens — first open takes ~2 seconds.
6. **Mark resolved** closes the break (kept in browser memory for this session). **Escalate via email** opens a modal with an LLM-composed email.

The app identifies files automatically by checking column headers, so the upload order doesn't matter.

## Authentication

The app is gated by a single password set in the `APP_PASSWORD` env var. The same value protects:
- The login overlay (frontend won't reveal the app until the password is correct)
- Every `/api/*` endpoint except `/api/login` itself (rejects with 401 if the bearer token is missing or wrong)

**To change the password:** update `APP_PASSWORD` in Vercel Settings → Environment Variables → Redeploy. Anyone with an old token will be bumped back to the login screen on their next API call.

**To revoke an existing session:** change the password. There's no per-session revocation in this PoC since there's only one user.

This is a **single-user gate**, not full auth. If you need per-user accounts, audit logs, or password reset flows, swap this for Clerk, NextAuth, or Supabase Auth.

## Cost expectations

Azure OpenAI pricing depends on your tenant agreement and the deployed model. For reference at standard Azure rates with GPT-4.1:
- 92 breaks classified per run ≈ **a few cents** total
- Per investigation draft ≈ fraction of a cent
- Per email composed ≈ fraction of a cent

A single user running ~10 reconciliations a day with full investigations should land **comfortably under $5/month** at standard rates. Set the Azure budget alert (step 4 above) and you're protected regardless.

To reduce costs further, swap `AZURE_OPENAI_DEPLOYMENT` to a smaller deployment like `gpt-4o-mini` if your tenant has one.

## Tuning

### Change the model
Set `AZURE_OPENAI_DEPLOYMENT` in Vercel env vars to the **deployment name** of any chat-completions-compatible model deployed in your Azure resource — e.g. `gpt-4.1`, `gpt-4o`, `gpt-4o-mini`. Note that this is the name *you chose* when deploying the model in Azure, not the model's family name.

### Use a different Azure API version
Set `AZURE_OPENAI_API_VERSION` to a version your deployment supports. The Azure portal lists supported versions per deployment. The current default is `2025-03-01-preview`.

### Adjust classification batch size
In `app.js`, search for `const BATCH = 12`. Smaller batches = more parallel requests = faster but more API calls. Larger batches = fewer requests but each one takes longer. 12 is a good middle ground for the 92-row Axiom file.

### Tighten or loosen the prompts
Each `/api/*.ts` file has its prompt as a top-level `SYSTEM_PROMPT` constant. Edit, redeploy, done.

### Add new break types
1. Edit the type enum in `api/classify.ts` `SYSTEM_PROMPT`.
2. Optionally add a tip for the new type in `app.js` → `tipForType()`.

## Limits to know about

- **Vercel Hobby function timeout: 10s.** This project sets `maxDuration: 30` in `vercel.json` — that needs the **Pro plan** ($20/month). On Hobby, lower it to 10 and reduce `BATCH` to ~6 if classifications time out.
- **Function body limit: 4.5MB on Hobby, 50MB on Pro.** The classify endpoint sends ~12 breaks per call (a few KB), nowhere near the limit. If your Axiom file ever exceeds 4.5MB compressed, you'd need Vercel Blob — but that's only at thousands of breaks.
- **No persistence.** Run history lives in browser `localStorage`. Clear browser data = lose history. Resolved-break state is per-session only. Add Vercel Postgres or Upstash when you need real state.
- **No actual emails sent.** The "Send" button is cosmetic. Wire up Resend, SendGrid, or Postmark when you're ready.

## What's not in this PoC (and where you'd add it)

- **Authentication** — add NextAuth or Clerk
- **Multi-user history** — add Vercel Postgres or Supabase  
- **Real email sending** — add Resend (5 lines of code in `api/send-email.ts`)
- **Audit log of agent decisions** — add a database; log every classification with input + output
- **Streaming responses for the agent log** — switch the API routes to Edge Runtime + Server-Sent Events; for a one-user PoC the current parallel-batch approach is simpler and the perceived latency is fine
